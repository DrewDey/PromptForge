## Iter 56 — 2026-04-17 — Purge "prompt" from step chrome vocabulary (Polish #1)

**What shipped.** `src/app/prompt/[id]/page.tsx` — the `StepContent` dispatcher rewritten so it owns the eyebrow vocabulary, with the `label` prop removed from its public API and from all four callsites. The dispatcher now keys off `detectContentKind(text)` (already in the codebase from iter 52) and the semantic `variant` prop to pick both the renderer AND the eyebrow label in a single place: prose-kind + prompt-variant → `<Prose label="ask">`; prose-kind + result-variant → `<Prose label="response">`; code-kind + prompt-variant → `<CodeBlock label="prompt">`; code-kind + result-variant → `<CodeBlock label="result">`. The literal developer vocabulary ("prompt" / "result") now survives only when the payload is actually code/schema/JSON that renders on the dark mono panel — everywhere else, the page speaks in conversational "ask" / "response" pairs. In lockstep, three pieces of inline detail-page copy were rewritten so the chrome outside the eyebrows matches: the Build-Path section subtitle "See what came out of each step. Expand to reveal the **prompt** behind it." → "…Expand to reveal the **ask** behind it."; the per-step disclosure `<summary>` "Show the **prompt** behind this step" → "Show the **ask** behind this step" (single location, 2026 lines of step rendering all route through it); the single-step (no-chain) branch's `<h2 class="text-xl font-black">The Prompt</h2>` → `<h2>The Ask</h2>` with a small block-comment above explaining the iter-56 vocabulary split for future readers. The inter-step "step N result → step N+1 prompt" chip at lines 430–438 was deliberately NOT touched — that chip is the subject of Polish #2 (a separate iteration that has to decide whether the chip survives at all), and rewriting its copy now would prejudge the reassessment. Callsites: 4 `<StepContent>` invocations (2 step-result, 2 step-prompt inside-the-disclosure + 1 step-prompt fallback when no result, 1 single-prompt branch) all drop the raw `label` string and rely on the dispatcher. `src/components/Prose.tsx` and `src/components/CodeBlock.tsx` are unchanged — the split was implementable purely inside the dispatcher.

**Why this approach (design rationale).** Backlog Polish #1 named three candidate prose labels — "ask" / "approach" / "brief" — and left the pick to the iteration. Chose **ask / response** over ask / approach / brief for a specific reason: the detail page is read as a conversation between an author and a model ("they said X, the model answered Y, then they said Z"). Conversational vocabulary (ask ↔ response) matches that mental model. "Approach" frames the entry as the author's strategy rather than their input; "brief" frames it as a formal spec deliverable. Neither feels right when the step payload is a casual, inline "write me three versions and explain the tradeoffs" question. Response pairs naturally with ask; result does not (result is a verdict, which is what Final Output at the top of the page now represents). On where the label-pick logic lives: considered three alternatives. (a) Keep the `label` prop on `StepContent` and have callsites pass the right string — rejected because that leaves four places in the file where a future editor could accidentally pass "prompt" as the prose label and undo the work; centralizing the pick means the vocabulary is enforced by type system + dispatcher, not by editor discipline. (b) Push the vocabulary split down into `Prose` itself (add an implicit default of "ask"/"response" based on variant) — rejected because `Prose` is a generic editorial card component (it's rendered once per detail page now, but could be reused in the Build page builder or in a future comments surface) and baking step-flow vocabulary into it makes that reuse awkward. The component stays vocabulary-neutral; the dispatcher is the vocabulary gatekeeper. (c) Keep labels on callsites but use a shared constants object (`CHROME_LABELS.prose.prompt = 'ask'`) — rejected as ceremony over substance; the dispatcher is already the single-source-of-truth surface this iteration needs, and a constants object would just be a second indirection. CHOSE the dispatcher-owned approach: `StepContent` takes `variant` (semantic role) and `text` and does the full lookup internally. This also means future classifier tweaks automatically route to the right label — if a natural-language payload flips to `code` after a heuristic tune, its eyebrow becomes "prompt" without anyone touching callsites. On the inline copy choices: "Expand to reveal the ask behind it" was chosen over "Expand to reveal what they asked" because "the ask" is a noun phrase that matches the eyebrow vocabulary pattern exactly — the subtitle is introducing the reader to the same word they'll see on the card. "Show the ask behind this step" over "Show the ask that produced this step": "behind" suggests the ask is the thing this step is built on (accurate — it IS the input payload), whereas "produced" implies causation the reader hasn't earned yet when the disclosure is still collapsed. For the single-prompt h2 "The Ask" (vs "The Approach"): the field below the h2 renders the author's literal natural-language input — calling that "The Approach" would suggest the author is explaining their methodology, but on the no-chain branch the value is usually a single direct question to the model. "The Ask" is honest about what follows.

**References consulted.** `WebFetch https://linear.app/changelog` succeeded and directly supported the choice: Linear's content uses utilitarian, direct vocabulary (Fixes, Improvements) rather than elaborate editorial framing, and the one place they use "Ask" it's a product noun (Linear Asks) — confirming that "ask" is read as **conversational/accessible** rather than jargony. The fetch report concluded: "ask/response reads as more conversational than technical, while prompt/result would feel more computational. Linear's tone favors accessibility over precision terminology." That's the exact brand direction ("Linear / Vercel / Raycast, modern dev tool") the CLAUDE.md pins as the target aesthetic — so ask/response aligns with the reference and stays tonally correct. No other fetch this iteration; the single reference carried the decision because the question was specifically about conversational editorial vocabulary (not visual/structural layout, where cross-referencing multiple sources matters more).

**Responsive / edge-case coverage.** The vocabulary swap is a rendering-path change, not a layout change, so breakpoint behavior is unaffected — every existing responsive rule on `<Prose>`, `<CodeBlock>`, the step card, and the single-prompt section continues to work verbatim. Edge cases explicitly verified: (1) chained project with all-prose payloads (3301, 3314) → 3 `ask` eyebrows + 3 `response` eyebrows per SSR pass, zero `>prompt<` or `>result<` eyebrow strays; (2) single-prompt project with no steps (3302, 3312) → `<h2>The Ask</h2>` renders once, Prose card beneath carries the `ask` eyebrow, zero legacy strings; (3) collapsed `<details>` disclosure → still in DOM, `ask` eyebrow sits inside it so screen readers and keyboard users accessing the collapsed surface get the new vocabulary without having to open the disclosure first; (4) a project where step content is literal code (e.g. a JSON schema the author pasted) → `detectContentKind` returns `code`, dispatcher routes to `<CodeBlock>` with label `prompt`/`result` — the literal developer vocabulary shows up exactly where the payload warrants it, so a future "here's the JSON I told Claude to match" step wouldn't get mis-labeled as "ask"; (5) meta string propagation → `meta={`step ${idx + 1}`}` still flows into both renderers unchanged, so the right eyebrow shows "ask · step 2" on Prose and "prompt · step 2" on CodeBlock. Focus/hover/interactive states: none of the interactive controls on the detail page were touched; the disclosure's `<summary>` hover ring, the CopyButton's ghost-fade-in on `<Prose>`, the CodeBlock's dark traffic-light header all carry through. Accessibility: the disclosure summary now says "Show the ask behind this step" — accurate for the usual prose-content case; if the underlying payload is code the summary phrasing is technically slightly imprecise ("ask" is conversational, "prompt" would be more literal) but the disclosure still correctly reveals the author's input, and the code variant's own eyebrow ("prompt") inside the revealed panel clarifies the payload type. Decided not to branch the summary text on kind — adding `const summaryText = kind === 'code' ? 'Show the prompt…' : 'Show the ask…'` would require hoisting `detectContentKind` up from the dispatcher into the page-level render loop purely to compose one string, and the 14/14 mock-data step payloads classify as prose (per iter 52's sanity test) so the code-content disclosure is a hypothetical not yet encountered on seeded projects.
