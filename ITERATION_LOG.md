(b) Keep a single-column layout and use a global `position: fixed; right: 24px; top: 96px` pinned panel on `lg+` — rejected because fixed positioning ignores container flow, so the panel would overlap the main content on narrower `lg` (1024–1200px) widths where the content column fills most of the viewport. (c) A left-rail instead of right-rail — rejected because PathForge's breadcrumb, title, and Final Output hero form the reader's entry funnel on the left side of the viewport; placing an action panel on the left competes with that reading anchor. Right-rail is the conventional action-surface location (GitHub PR, Vercel dashboard, Linear issue detail). ## Iter 54 — 2026-04-17 — The Story: callout → editorial feature body (Polish #5)

**What shipped.** `src/app/prompt/[id]/page.tsx` — The Story section rewritten from a bordered documentation-callout to un-framed editorial prose. Removed the `bg-primary-50/60 border-l-4 border-brand-orange p-6 sm:p-8` callout block and the `MessageSquare` icon + `<h2 class="text-xl font-black">` heading combination that together read as "alert! important note!". Dropped the `MessageSquare` import entirely (no remaining consumers). Added a `storyParagraphs` derivation before the return block that splits `prompt.content` on `\n\s*\n` (blank-line boundaries), trims, and filters empties so the author's paragraphing becomes real `<p>` elements. Now renders a quiet mono eyebrow row — `bg-brand-orange` 1.5×1.5 dot + `h2#story-eyebrow` with `text-[11px] font-mono uppercase tracking-[0.18em] text-surface-600 font-semibold` reading "The story" + a `·` divider + `text-surface-400` meta "why they built it" — explicitly matching the eyebrow vocabulary iter 51 introduced on the Final-output hero so the two sibling blocks read as a set rather than as two competing treatments. Body is `max-w-prose` (≈65ch) containing mapped `<p>` elements at `text-lg sm:text-xl leading-[1.75] text-surface-800` with `whitespace-pre-line` (preserves intra-paragraph line breaks the author typed). The FIRST paragraph also carries `first-letter:font-black first-letter:text-[3.5rem] sm:first-letter:text-[4rem] first-letter:text-brand-orange first-letter:mr-1.5 first-letter:float-left first-letter:leading-[0.85] first-letter:mt-1.5` — a drop-cap treatment that uses the real CSS `::first-letter` pseudo-element (Tailwind's `first-letter:` variant), so the rest of the paragraph text naturally flows around the enlarged glyph. The h2 stays in the DOM (a11y outline + `aria-labelledby` landmark) but its visual presentation is demoted to eyebrow weight. Subsequent paragraphs use the same body type scale with `mt-5` separation and no drop cap — feature-article body continues. Empty `prompt.content` → section renders nothing (the whole block is gated on `storyParagraphs.length > 0`), so projects without a story don't leave a floating eyebrow.

**Why this approach (design rationale).** Backlog #5 offered two directions: "generous editorial prose, no box" or "magazine intro with inline author avatar + drop-cap first character." I picked a hybrid leaning strongly toward the first, with the drop cap as the one concession to intro-block theatre. The author avatar/byline was already rendered in the page header (lines 142–159) — pulling a SECOND avatar into The Story would have been redundant and would also have introduced a competing identity-anchor on the page. Keeping the byline in the header, where it's a proper page-level credit, and letting The Story read as pure author voice under a muted eyebrow is more editorial and less newsletter-masthead. Considered three alternatives for the frame treatment. (a) Keep a faint background wash (e.g., `bg-surface-50`) to preserve block-ness — rejected because surface-50 is `#fafafa`, identical to the page body, so it'd be a box without a visible border, which is just a shape with none of a shape's value. (b) Render a left-side thin rule (`border-l border-surface-200 pl-6`) as a quiet editorial flourish — rejected because the step-card section below already uses a vertical gradient pipe on the left, and a second left-rail element above it would read as "same vocabulary, lower weight" instead of "different block." (c) Fully un-framed prose on the page background, relying on the eyebrow and the drop cap to signal block boundaries — CHOSE THIS. Linear's changelog (WebFetch reference) explicitly validates the pattern: "No colored boxes, left borders, or alert-style callouts… [relies on] substantial vertical breathing room." My mb-14 (56px) below The Story and the eyebrow-above-body structure mirror Linear's "timestamp → headline → body" cadence. The drop cap diverges from Linear deliberately — Linear's changelog entries are short and don't need an eye-anchor, but PathForge's Story sits between a heavy gradient-framed hero above and a heavy dark-bar-headed step card below, and the reader needs a clear "now we're into the narrative" hook. A drop cap does that without reintroducing any box or border. Scale selection (`text-[3.5rem]` / `sm:text-[4rem]`) is deliberate: smaller than typical magazine drop caps (which run 80–100px) because the page body is 18–20px, and the ratio needs to be ~3× body size to read as intentional rather than accidental. `leading-[0.85]` + `mt-1.5` pulls the glyph's baseline to align with roughly the second text line — CSS `::first-letter` baseline defaults leave the glyph floating too high relative to the flowing text. On split strategy: considered splitting on `\n` (any newline) — rejected because mock-data and seed authors use single `\n` for intentional line breaks within a thought, and splitting on any newline would fragment those. Double-newline is the universally understood paragraph boundary (Markdown, plain text, email). `whitespace-pre-line` inside each `<p>` preserves intra-paragraph breaks authors added.

**References consulted.** `WebFetch https://linear.app/changelog` succeeded and directly informed the un-framed choice: "No colored boxes, left borders, or alert-style callouts frame the opening content"; "body flows naturally without special formatting"; "generous margins surround content blocks. This prevents a cramped, documentation-heavy sensation." `WebFetch https://dev.to/top/infinity` returned the article-list feed only (no body styling visible from the listing page) — attempt noted, didn't cite specifics. `WebFetch https://stripe.press/pdfs/the-pmarca-blog-archives.pdf` returned HTTP 404 (URL dead) — attempt noted, didn't cite. Two of three attempts fetched; Linear's confirmed the core pattern and the drop-cap divergence is documented as a deliberate deviation above. The eyebrow vocabulary is borrowed directly from iter 51's Final-output hero, which was itself grounded in the editorial-gallery style (Vercel templates + Linear changelog per that iteration's log).

**Responsive / edge-case coverage.** Desktop (sm+) uses `text-xl` body + `text-[4rem]` drop cap; mobile uses `text-lg` body + `text-[3.5rem]` drop cap — proportionally matched (both ~3.5× body size). Parent container already provides `px-4 sm:px-6 lg:px-8`, so the max-w-prose body never touches the viewport edge. `float-left` on the drop-cap works across all modern browsers and degrades gracefully — if a legacy browser strips `::first-letter` entirely, paragraph renders as plain body text (no broken layout). `mt-14` (mb-14) separates The Story from the hero above and the Build Path below at `56px` — matches the detail page's existing section-rhythm (`mb-10` / `mb-12` sibling spacing). Edge cases explicitly handled: (1) `prompt.content` empty/null → entire section elides (`storyParagraphs.length > 0` gate); (2) single paragraph → only renders one `<p>`, drop cap fires, no subsequent paragraphs; (3) many paragraphs → each gets `mt-5` separation after the first, `max-w-prose` caps measure; (4) paragraph starts with an opening quotation mark or em-dash → CSS `::first-letter` correctly selects the first letter AND its leading punctuation per spec, so `"The idea` → drop cap is `"T` styled together (this is the standard typographic treatment — intentional); (5) very long first paragraph → drop cap floats left while text wraps around it correctly because the `float-left` + `leading-[0.85]` keeps the glyph height to ~2 lines of body text; (6) intra-paragraph author line breaks → `whitespace-pre-line` preserves them without fragmenting the paragraph semantically. Focus/interactive state: none needed — this is pure text content, no interactive elements. Accessibility: `h2#story-eyebrow` landmark is kept (`aria-labelledby="story-eyebrow"` on the section), so screen readers still get a proper section heading even though the visual weight is eyebrow-scale. The hidden `·` separators use `aria-hidden="true"`.
