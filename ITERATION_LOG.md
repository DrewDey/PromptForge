## Iter 52 — 2026-04-17 — Demote CodeBlock for natural-language step content (Polish #1)

**What shipped.** Added a new sibling `<Prose>` component (`src/components/Prose.tsx`) and a new classifier (`src/lib/content-kind.ts`) that auto-selects between CodeBlock (dark mono panel) and Prose (light editorial card) per step payload. The detail page (`src/app/prompt/[id]/page.tsx`) now routes every step prompt, step result, and single-prompt-branch through a `<StepContent>` dispatcher that calls `detectContentKind` and picks the right renderer. `<Prose>` shape: white card with hairline `border-surface-200`, quiet eyebrow row (accent dot + small-caps `tracking-[0.16em]` label + optional meta), reading-weight sans body at `text-[15px] leading-[1.7] text-surface-800` with `whitespace-pre-line` and `max-w-prose`, and a ghost `CopyButton` (new third variant) positioned `absolute top-2.5 right-2.5` that fades in on hover/focus. Heuristic signals that flip to `code`: presence of ` ``` ` fence, HTML-ish tags >25% of lines, `;` line-endings >30%, brace-only lines >15%, or dense indentation >50% of lines (only when ≥4 lines). Defaults to `prose` — step content on PathForge is overwhelmingly natural language (14/14 mock-data step payloads classify as prose, verified via an inline node sanity test). Also added a third `ghost` variant to `CopyButton` (white-on-white with hairline border, label hidden at rest, appears on `copied` state) so Prose's copy affordance doesn't compete with the body text.

**Why this approach (design rationale).** Backlog offered a choice: overload CodeBlock with a `naturalLanguage` prop, or ship a sibling `<Prose>`. Chose the sibling. CodeBlock's name carries semantic weight — "code" in the identifier tells the next reader what it renders. A CodeBlock in natural-language mode would be a branchy component where half the codepaths don't render as code, making callsites harder to reason about and defeating the documentation value of the name. A sibling forces the callsite to declare intent, and `<StepContent>` — a tiny dispatcher — centralizes the selection logic so no future detail page picks the wrong component. I also considered rendering Prose with a light surface color (`surface-50` or faint orange tint) instead of pure white. Rejected: the step card already has a `bg-surface-900` header bar and sits on the page's white body background; a tinted Prose inside a white-bodied card would introduce a third subtle-surface tone in the same 400px viewport stretch. Pure white keeps the Prose panel legible as "content, not chrome." On copy-button placement: tried putting it in the eyebrow row inline next to the meta; rejected because the eyebrow is quiet-by-design and a button at rest there pulls the eye from the body. Absolute-positioned ghost-fade lets the prose breathe by default and surfaces the action exactly when the reader's cursor is already in the block. On the label prop: kept "prompt"/"result" for the iter-52 scope — renaming to "ask"/"approach" is Polish #3 in the queue, explicitly a separate iteration. Shipping both at once would couple a component refactor with a copy rewrite and make the log less useful to future-me. Heuristic tuning: the dense-indentation signal originally fired at >30% of lines, which misclassified a prose block with a bulleted list ("    • one", "    • two", ...) as code. Bumped to >50% AND requires ≥4 lines — short prose with a trailing indent doesn't fire. The node sanity test covers that case (`Mostly natural language but the second line happens to be indented a bit.` → prose).

**References consulted.** `WebFetch https://linear.app/changelog` succeeded and confirmed the pattern: Linear presents long-form prose "without explicit framing — long-form text blocks lack visible borders or background colors… [relying on] typographic hierarchy, whitespace, and subtle semantic differentiation." My Prose diverges deliberately by keeping a hairline border, because PathForge's detail page sits multiple Prose blocks directly adjacent inside a step card (prompt + result, then prompt + result of the next step), and totally un-framed blocks would read as one continuous paragraph rather than as distinct contributions. A hairline surface-200 border is the minimum edge-weight that preserves "block-ness" without competing with content. `WebFetch https://read.cv` returned HTTP 402 (bot/paywall block) — attempt noted, didn't cite. `WebFetch https://docs.stripe.com/` returned only navigation chrome — attempt noted, didn't cite. Two of three references fetched; SKILL's floor (don't hard-halt if all three fail) wasn't reached. On the ghost CopyButton: Raycast's command-row "copy" affordance surfaces on hover and hides at rest — that's the behavior I borrowed (opacity-0 → opacity-100 on `group-hover` and `focus-within`).

**Responsive / edge-case coverage.** Prose padding is `px-5 pt-2.5 pb-5` — narrow mobile stays readable without breakpoint variants (no `sm:px-8` jump that'd introduce a visual shift at 640px). Body font size at `text-[15px]` is a deliberate notch below CodeBlock's `text-[13px]` — prose reads longer than monospace so needs more x-height; 15px is the site's Linear-like editorial-body size (matches the PromptCard description). `max-w-prose` caps the measure at ~656px so a very long result on a wide step card doesn't read as a wall. `whitespace-pre-line` preserves author line breaks; `min-w-0` not needed because the parent step-card already clips overflow. Focus handling: the `CopyButton` ghost variant retains the site-standard `outline-brand-orange focus-visible` ring (inherited from the variant switch in CopyButton.tsx), and the Prose wrapper uses `focus-within:opacity-100` so keyboard-focusing the button reveals it even without a pointer hover. Edge cases handled explicitly:
- Empty/whitespace-only content → detector returns 'prose' (empty text is vacuously prose; CodeBlock would render an empty dark panel which reads worse).
- Markdown-fenced code (e.g. a JSON blob inside a prose explanation) → flips the whole payload to `code` because inside a fence is the strongest "treat me as literal" signal; the prose explanation around it is a small price for the fence's hard guarantee.
- Mixed-kind payloads (prose with inline `if (x) { y }`) → single occurrences don't trip the detector because every signal uses a ratio against total line count, not absolute count.
- Short prose (1–3 lines) → dense-indent signal is gated behind ≥4 lines so tightly-formatted short prose doesn't false-flag.
- Details/disclosure container already existed from iter 49; Prose renders correctly inside the nested `<details>` (the open/close toggle is on an outer `group` wrapper, Prose has its own `group/prose` for its own hover interactions — no collision because of the slash-namespaced Tailwind group syntax).

**Verification scope.** `npx tsc --noEmit` passes clean. `npm run build` passes clean — all 10 static pages generated, TypeScript pass in 1.76s. Inline Node sanity test (`/tmp/kind-test.mjs`) runs 9 hand-authored cases (prose w/ paragraphs, prose w/ numbered list, fenced JSON, JS function w/ semicolons, HTML fragment, bare JSON, short prose w/ single indented line, all-indented block) — 9/9 pass. Mock-data scan (`/tmp/kind-mock.mjs` against `src/lib/mock-data.ts`) — extracted 14 template-literal `content:` / `result_content:` fields, 14/14 classify as prose, 0/14 as code. This is the expected distribution for PathForge's current seed data; validates that the common case goes to Prose without needing per-field author overrides. Chrome MCP verify attempted — `tabs_context_mcp` created a new tab (id 1834517331) but reported URL stuck on `chrome://newtab/` after two `navigate` calls, and `read_page` / `javascript_tool` returned "Cannot access a chrome:// URL" on both attempts. The separate `npm run build` was the backup path and completed clean.

**Follow-ups spotted.**
- Polish #3 in the queue was already pointing at renaming `label="prompt"`/`"result"` to `"ask"`/`"approach"` on the prose path. Now that Prose is live, that rename is the natural next iteration — and I've updated the backlog entry to call out Prose specifically.

If a third surface ever needs the same prose-vs-code auto-routing (builder preview rail already exists for the form; user-profile cards might eventually too), extract it to `src/components/StepContent.tsx`. Not worth doing speculatively for one callsite.
- Heuristic is conservative: a prose block that quotes a single curly-braced JSON literal (`{"key": "value"}`) on its own line can trip the brace-only detector once that one line exceeds 15% of total lines in a short payload. ## Iter 51 — 2026-04-17 — Case-study "Final output" hero above The Story (Polish #1)

**What shipped.** Inserted a new `<section aria-labelledby="final-output-eyebrow">` between the fork CTA (line 174 of `src/app/prompt/[id]/page.tsx`) and The Story. The hero resolves `heroContent = prompt.result_content?.trim() || lastStepWithResult?.result_content?.trim() || null` and renders only when a substantive output exists. Frame: a hairline gradient border — `bg-gradient-to-br from-brand-orange/55 via-surface-200 to-brand-blue/55 p-[1.5px]` around an inner `bg-white px-6 py-8 sm:px-10 sm:py-12 min-h-[220px] flex flex-col` card. Eyebrow is "Final output" with a blue 6×6 dot in mono/uppercase/tracking-[0.18em] style, followed by a `·` separator and a meta — either `what they shipped` (when sourcing from `prompt.result_content`) or an in-page anchor link `from step N of M` (when falling back to the last step's `result_content`). The body paragraph renders at `text-lg sm:text-xl text-surface-900 leading-[1.55] whitespace-pre-line max-w-prose` — no `CodeBlock`, no mono, no dark-editor chrome. Legacy bottom "The Result" section (was `border-l-4 border-brand-blue` callout at old line 363–374) removed entirely since it was double-rendering the same payload. Also dropped the `ArrowDown` import usage at the old Result H2 (still used inter-step, so kept in the import list). The Story keeps its orange callout — intentional, so the hero's editorial-gradient card doesn't compete with a parallel editorial box right below it (Polish #7 will revisit The Story's callout).

**Why this approach (design rationale).** The backlog entry offered two directions: "a dark panel with subtle orange→blue gradient border, or a generous editorial card." Considered both. Rejected the dark panel because the page already carries heavy dark weight — the header strip is `bg-surface-900`, every step card has a `bg-surface-900` header bar, and every `CodeBlock` in the Build Path is a dark mono panel. Adding a third dark surface above The Story would read as yet another "code thing" and defeat the "this is a build showcase, not a prompt library" vision anchor. The editorial white card with a hairline gradient edge is the opposite move — it borrows the site's orange→blue spine vocabulary (same gradient already lives in the vertical step pipe and the journey strip from iter 50) but applies it as a picture-frame edge rather than a surface fill, keeping the content light-on-white and reading as a magazine feature's primary exhibit. Also considered putting the hero INSIDE the existing `header` block right under the pills (tighter coupling to the title group). Rejected because then the fork CTA gets visually buried behind the hero, and the CTA is the single highest-intent button on the page per backlog Polish #3. Placing the hero *after* the CTA keeps the "read title → scan metadata → hit fork OR keep scrolling → see the output" flow intact. Fallback meta wording matters too: the `from step N of M` link is a real `<a href="#step-N">` so a reader can jump to where the output came from in the chain — this is the opposite of hiding the source, it's making the chain navigable from the hero.

**References consulted.** `WebFetch https://read.cv` returned HTTP 402 (paywall/bot block) — attempt noted, couldn't cite. `WebFetch https://linear.app/changelog` worked: confirmed the pattern I wanted — hero visual sits below title/subtitle with NO visible border or frame, "clean, minimal styling with slight drop shadows for depth," body prose follows. I diverged from the no-border path intentionally: Linear's pattern assumes a literal screenshot or video where the content IS the visual. PathForge's "final output" is usually prose, so an unframed block would just look like a floating paragraph and disappear. A hairline gradient frame gives the prose block enough edge-weight to read as "an exhibit" without fighting the content inside. `WebFetch https://vercel.com/templates` returned a generic description (their HTML structure didn't expose CSS details), so I couldn't cite specific treatment — noted, not load-bearing for this choice. The shadow I did add — `shadow-[0_1px_2px_rgba(15,23,42,0.04)]` — is the Linear "slight drop shadow for depth" move, restrained.

**Responsive / edge-case coverage.** Hero uses `px-6 py-8 sm:px-10 sm:py-12` so mobile gets 24×32px padding and desktop 40×48px — scales without breakpointing the frame itself. Body text steps `text-lg` → `sm:text-xl` so mobile reads at 18px and desktop at 20px. Eyebrow row uses `flex flex-wrap items-center gap-x-2.5 gap-y-1.5` so the "Final output · from step 3 of 3" meta wraps to a second row on very narrow widths rather than pushing off-screen. `whitespace-pre-line` preserves the author's intentional line breaks in `result_content` (mock data uses explicit `\n` formatting throughout). Focus-visible outline on the step-link fallback (`hover:text-brand-blue transition-colors duration-200` — matches the site's blue-is-result vocabulary). Edge cases handled:

The step card *also* shows the same payload inside its `CodeBlock`, but that's acceptable: the hero positions it above-the-fold as editorial prose; the step view keeps it as the formal result within the chain. Two contexts, two treatments, one payload.
- Project has neither → hero doesn't render (whole `{heroContent && (...)}` guard), page goes straight from fork CTA to The Story as before.
- Whitespace-only `result_content` → `?.trim() || null` rejects it, falls back or hides.

**Verification scope.** `npx tsc --noEmit` clean. `npm run build` clean — full production build succeeded, all 10 static pages generated, TypeScript typecheck passed. Chrome MCP on `/prompt/...3314` (3-step project with top-level result): accessibility tree confirms order `banner (title/pills/fork CTA) → region [ref_64] (Final output + what they shipped + body) → region [ref_68] (The Story) → region [ref_71] (Build Path)` — hero renders between fork CTA and The Story exactly as intended; legacy bottom "The Result" heading absent from a11y tree (successful removal).
